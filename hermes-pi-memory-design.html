🎯 Executive Summary

The video dissects AI agent memory into three jobs — Storage, Injection, and Recall — then cherry-picks the winning approach from three frameworks for each job:

Job Framework Borrowed From Key Idea
Storage MemSearch Auto-capture after every turn, summarized by a cheap model, appended to a daily log. No agent opinion on what matters — everything gets captured but condensed.
Injection Hermes (frozen snapshot) Identity + user profile + recent memories injected at session start. Capped at ~1,300 tokens. Cached once per session.
Recall MemSearch (hybrid search) + GBrain (reranker + cited answers) Multi-tier: check injected memory first → hybrid semantic/keyword vector search → reranker → written answer with citations. Admits when it doesn't know.
💡 What This Means for Hermes + Pi Hermes already has frozen-snapshot injection (MEMORY.md/USER.md). Pi has an extension system for dynamic context. The composite approach fills each agent's gaps: Hermes needs auto-capture + semantic recall, Pi needs the whole system built via its extension API. A shared vector store bridges both.

Multi-Tier Recall Flow (from the video)

User asks about X │ ├─ TIER 0: Check injected memory (frozen snapshot in context) │ └─ Found? → Answer directly from context (fastest, no API calls) │ └─ Not found? → Continue │ ├─ TIER 1: Vector hybrid search (semantic + keyword BM25) │ └─ Query local vector DB with user's question │ └─ Returns top-10 chunk candidates with scores │ ├─ TIER 2: Reranker (cross-encoder second pass) │ └─ Reorder candidates by relevance │ └─ Keep top-3 to top-5 │ └─ TIER 3: Synthesize cited answer (GBrain pattern) └─ LLM writes answer with source citations └─ "I don't know" if nothing relevant found └─ Always cite: "[source: 2026-05-12_daily.md#decision-3]"

  • Phase 2 — Hermes Enhancement Hybrid Search + Cited Answers

🔨 4. Phase 1 Deep Dive: Foundation

4.1 Vector Store Decision

OptionProsConsRecommendation
sqlite-vec
(asg017/sqlite-vec)		 Zero infrastructure. Same DB format as Hermes already uses. Embeddings stored inline. Works offline. SQLite FTS5 for BM25 keyword search natively. Single-node (not a problem — you only need local). ✅ The one and only choice
Perfect for single-user local use
⚠ SQLite Concurrency Three processes write to the same sqlite-vec DB (Hermes sidecar, Pi extension, cron evaluator). Without safeguards, concurrent writes cause SQLITE_BUSY — captures are lost, queries return stale data. Required from day one:
  • WAL mode — PRAGMA journal_mode=WAL; at DB creation. Allows concurrent reads during writes.
  • Write retry — CLI implements exponential backoff on SQLITE_BUSY (3 retries, 50ms base delay).
  • Busy timeout — PRAGMA busy_timeout=5000; (5 second wait before failing).
  • Single writer convention — The cron evaluator is the only batch writer; capture and query are fast single-row ops.

4.2 Local Embedding Model

The video emphasizes on-device, zero API cost vectorization. Recommend:

ModelDimensionQualitySpeedPackage
all-MiniLM-L6-v2384GoodVery fastsentence-transformers
BGE-small-en-v1.5384BetterFastsentence-transformers
BGE-base-en-v1.5768BestModeratesentence-transformers
💡 Recommendation: BGE-small-en-v1.5 (384d) Good balance of quality/speed for a local agent. Embeddings can be recomputed lazily. Runs on CPU on modern Macs in ~20ms per sentence. If you later add a GPU Droplet, swap in BGE-base or intfloat/e5-mistral-7b-instruct for higher quality.

4.3 Reranker Model

For Tier 2 (reranker pass), use a cross-encoder. This is the most expensive inference step but only runs on Tier-1 candidates (top 10 → rerank 3-5):

ModelPackageLatency (CPU, Mac)
cross-encoder/ms-marco-MiniLM-L-6-v2sentence-transformers~50ms per pair
BGE-reranker-v2-m3FlagEmbedding~100ms per pair (higher quality)

4.4 Auto-Capture Architecture

After every conversation turn: │ ├─ 1. Capture: Collect all new user + assistant messages since last capture │ (tracked by session_id + last_captured_message_id) │ ├─ 2. Summarize: Send to OpenCode Go API │ POST https://opencode.ai/zen/go/v1/chat/completions │ Authorization: Bearer ${OPENCODE_GO_API_KEY} │ Model: configurable (e.g. minimax-m2.5 or deepseek-v4-flash) │ Prompt: "Summarize the new information, decisions, facts, and │ preferences discussed in this exchange. Be concise." │ Expects: 2-4 bullet points max │ Cost: ~$0.0002/call at current model pricing │ ├─ 3. Tag: Extract named entities + auto-tags │ (project names, client names, technical topics) │ Optional — dedicated NER model or LLM call │ ├─ 4. Embed: Compute vector embedding via local model │ └─ 5. Store: Write to vector store content = summary text embedding = vector source = "hermes" | "pi" timestamp = now tags = ["project:jarvis", "client:uncle-reds", ...]
✅ Why OpenCode Go for Summarization No extra provider, no extra API key, no extra billing relationship. The hpm CLI calls the same endpoint Hermes already uses:
  • Base URL: https://opencode.ai/zen/go/v1
  • Auth: OPENCODE_GO_API_KEY (already in your .env)
  • Model: configurable — use minimax-m2.5 (fastest/cheapest) or deepseek-v4-flash (same as your main model)
  • Zero additional setup. One HTTP call.
The summarization model is configurable per environment (dev vs. prod) so you can use a cheap model for daily capture and a stronger one for spot-check evals.

4.5 Capture Frequency Trade-offs

StrategyGranularityStorageRecall QualityCost
Every turn (video's choice)FineHigher (but summarized)Best — nothing missedHighest — cheap model per turn
Every N turnsMediumMediumGood — may lose contextLower
End of session onlyCoarseLowestWorst — long sessions lose middleCheapest
On explicit /rememberUser-drivenMinimalLeaky — user forgets to save~Free (tool call)
✅ Recommendation: Every-turn capture, immediate embedding Follow the video's pattern: after each turn, summarize via the OpenCode Go API (e.g. minimax-m2.5 or deepseek-v4-flash) and embed immediately using the local BGE-small model (~20ms on CPU). The embedding cost per capture is negligible — deferring batch embedding until the next cron run would leave memories invisible to semantic recall for hours, which defeats the purpose of a memory system. The daily log at ~/.hermes/memories/daily/2026-06-16.md is kept as a text backup and audit trail, not as the primary recall source.

⚠ Integration Risk Hermes doesn't expose a native "post-turn hook" in its plugin API (currently plugins add tools, slash commands, and lifecycle hooks but not mid-conversation event hooks). The auto-capture will need to run as a background sidecar process that watches ~/.hermes/state.db (the SQLite session store) for new messages, captures them, summarizes, and stores. This is architecturally clean but means the capture process isn't embedded in Hermes' event loop — it's an independent observer.

6.2 Pi Extension Sketch

// pi-memory-extension.ts — rough design

import { Extension, Tool, Command } from '@earendil-works/pi-coding-agent';

export class MemoryExtension implements Extension {
  name = 'pi-memory';
  version = '0.1.0';

  tools: Tool[] = [
    {
      name: 'memory-find',
      description: 'Search past memories with semantic understanding',
      handler: async (query: string) => {
        // Call shared Python CLI: `hpm query "..." --limit 5`
        // Parse results, return with citations
      }
    },
    {
      name: 'memory-save',
      description: 'Explicitly save a fact to memory',
      handler: async (content: string, tags?: string[]) => {
        // Call shared Python CLI: `hpm save "..." --tags ...`
      }
    }
  ];

  // If Pi exposes a post-turn hook:
  onAfterTurn?: async (turn) => {
    const summary = await cheapLLM.summarize(turn.messages);
    await saveToVectorStore(summary, { source: 'pi', ... });
  };

  // Inject relevant context at session start:
  augmentContext?: async () => {
    const recentMemories = await queryVectorStore('', { limit: 5 });
    return `## Relevant Memories\n${recentMemories.map(m =>
      `- ${m.content} [source: ${m.timestamp}]`
    ).join('\n')}`;
  };
}

The CLI lives in one place and both agents call it. This avoids duplicating the memory logic in two codebases.

🤝 7. Phase 4: Cross-Agent Coherence

7.1 Shared Memory Challenges

ChallengeSolution
Both agents save contradictory facts Timestamp wins — newer fact supersedes. Old fact is archived (not deleted) with a superseded_by pointer. The cited-answer synthesis includes a conflict note: "Note: this was updated on 2026-06-14..."
Duplicate entries Content-dedup via SimHash on summary text before storing. Matches within 0.85 semantic similarity are merged (update timestamp, append source).
Context window pollution Injection is capped at ~1,300 tokens (video's recommendation). The injected content is dynamically selected: top-3 most relevant memories + frozen profile + essential facts, not everything.
Privacy / scope Each memory entry is tagged with access_scope. Hermes sees all. Pi extension filters by scope if configured.

7.2 Deduplication Flow

Before insert: │ ├─ 1. Compute embedding of new content │ ├─ 2. Search for nearest neighbor (cosine similarity > 0.85) │ └─ No match → INSERT (new entry) │ └─ Match found → │ ├─ Update timestamp to now │ ├─ Append source to source array ["hermes", "pi"] │ ├─ If content differs significantly, store as variant │ └─ Don't create duplicate row │ └─ 3. Return success

📊 8. Phase 5: Decay Evaluator & Dashboard

5.1 Decay Model

From the context-engine-design skill at ~/code/context-engine/bin/decay_evaluator.py:

score = 0.5 ^ (hours_since_reinforcement / half_life)
ParameterDefaultMeaning
half_life168 hours (1 week)How long before a memory's score halves without reinforcement
eviction_threshold0.25Below this → flagged for potential eviction
reinforcementOn access / matchEvery time a memory is retrieved in a query, its score resets to 1.0 (reinforced)

5.2 Spot-Check Loop (Silent Feedback Loop Mitigation)

Pure decay models have a blind spot: a stale entry keeps being retrieved, the user doesn't correct it, and absence of correction looks like reinforcement. Mitigation — run once per day as part of the daily cron cycle, sampling the 5 lowest-scoring entries:

Every evaluation cycle: │ ├─ 1. Sample the 5 lowest-scoring entries │ (most likely to be stale) │ ├─ 2. Send to LLM for spot check: │ "Rate each entry as STALE / QUESTIONABLE / VALID / INSUFFICIENT" │ ├─ 3. Adjust score: │ STALE → score -= 0.3 │ QUESTIONABLE → score -= 0.1 │ VALID → score += 0.05 │ (clamped to [0.0, 1.0]) │ ├─ 4. Log reasoning for auditing │ └─ 5. The LLM never directly evicts — only adjusts scores

5.3 Dashboard

Self-contained HTML (Chart.js from CDN, no server needed) with:

  • Score-over-time line chart with threshold line + spot-check markers
  • Distribution histogram — how many entries at each score level
  • Full entries table with filters (source, tags, status)
  • Most-recently-reinforced and oldest-unreinforced lists

🚀 10. Recommended First Step

✅ Start Here: Phase 1, Chunk 1 Build sqlite-vec + hpm CLI + auto-capture sidecar + immediate-embedding demo.

Scope: A single session, ~3-4 hours:

  1. Set up sqlite-vec with WAL mode — install extension, create DB with WAL, busy timeout, and the full schema
  2. Install BGE-small embedding model — pip install sentence-transformers + verify on a test sentence
  3. Build hpm capture — takes raw text, calls OpenCode Go API (configurable model, e.g. minimax-m2.5) to summarize, embeds immediately with local model, stores with retry
  4. Build hpm query — hybrid search (sqlite-vec ANN + FTS5 BM25), returns top-5 with scores
  5. Build hpm save — direct fact save for explicit /remember
  6. Build Hermes auto-capture sidecar — watches ~/.hermes/state.db for new messages, pipes them through capture
  7. Demo: Run a conversation in Hermes, let the sidecar capture it, then query it back with a different wording

Acceptance criteria:

  • A memory saved as "switched from Stripe to Paddle for Japanese market" is found with query "what payment processor did we pick for Japan"
  • Result includes a source timestamp and session reference
  • The Hermes sidecar is running, live-captures a turn, and the memory is queryable within seconds (not hours)
  • Hermes can call the CLI from a terminal command
💡 Why This First This validates the highest-risk unknowns in one shot:
  • Does BGE-small provide good enough semantic recall for your domain?
  • Does sqlite-vec with WAL mode handle concurrent writes cleanly?
  • Can the cheap model produce summaries that preserve decision context?
  • Does the Hermes state.db sidecar reliably capture every turn?
All must work before any agent integration is worth building. If this demo fails, the architecture needs different choices — and it's better to discover that before writing extension code.
